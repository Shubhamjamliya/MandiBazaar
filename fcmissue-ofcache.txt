The issue was a rare "Double Instance" module caching bug caused by mixing static and dynamic imports.

Here is exactly what happened step-by-step:

When you start your backend, server.ts statically imports firebaseAdmin.ts at the top of the file.
server.ts calls initializeFirebaseAdmin(). This successfully connects to Firebase and sets a variable inside that file called firebaseInitialized = true.
However, inside your fcmTokenRoutes.ts file, the notification sender was being loaded dynamically using this line: const { sendNotificationToUser } = await import("../services/firebaseAdmin");
Here is where it broke in production: In your local environment, Node.js is smart enough to realize both imports point to the exact same file, so it shares the memory between them. But in your compiled production environment, Node treated that dynamic import() as a request for a brand new, separate file!

This created two separate copies of firebaseAdmin.ts in your server's memory:

Copy A (Server): Initialized perfectly and connected to Firebase (firebaseInitialized = true).
Copy B (Routes): A completely fresh, uninitialized copy where firebaseInitialized was still false!
So when you clicked the Test button, it used Copy B, saw that it wasn't initialized, and threw the error.

By changing the dynamic import() to a normal static import at the very top of fcmTokenRoutes.ts, it forces the production server to share Copy A across your entire app, completely fixing the bug!
